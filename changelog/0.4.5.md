# v0.4.5 — AST dump parity fixes

Six AST dump correctness bugs are fixed, bringing corpus-astdiff from ~4% mismatch
rate down to near zero. Fixtures 115-118 cover the new cases.

## Fixes

- **`Module()` for empty files**: `Module(body=[])` is now `Module()` under
  Python 3.13+ show_empty=False mode, matching CPython's ast.dump output for
  empty source files.

- **`Try()` empty handlers omitted**: `Try(body=..., handlers=[], ...)` now
  omits `handlers=[]` when the handlers list is empty and show_empty=False,
  matching CPython. A try/finally with no except clauses no longer emits the
  empty handlers field.

- **f-string constant merging**: When two or more f-strings (or a mix of
  f-strings and plain strings) are implicitly concatenated, adjacent literal
  parts at the boundaries are now merged into a single `Constant` node,
  matching CPython. Three merging paths were fixed:
  - Between two adjacent f-strings: `f"a{x}" f"b"` → one Constant `ab` not two.
  - Trailing plain string after f-string: `f"a{x}" "b"` → merged.
  - Buffered plain string between two f-strings: `f"a{x}" "b" f"{y}"` → merged.

- **f-string escape sequences**: The f-string text segment scanner was missing
  `\a`, `\b`, `\f`, `\v`, `\xHH`, `\uHHHH`, `\UHHHHHHHH`, and octal escapes.
  These are now processed correctly, so `f"\x1b[{n}m"` produces a Constant
  containing the actual ESC character (0x1b), not the literal 4-char sequence.

- **Complex imaginary formatting**: `123456789j` was dumped as `1.23456789e+08j`
  due to Go's `'g'` format. It is now formatted using `pyFloat` with the
  trailing `.0` stripped for integer-valued imaginary parts, matching Python's
  `repr(123456789j)` → `'123456789j'`.
