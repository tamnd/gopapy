# v0.6.2: ASCII character classification lookup tables

The identifier scan loop in `parser/lex.go` now uses 128-entry byte-indexed
lookup tables instead of `utf8.DecodeRuneInString` + `unicode.IsLetter` for
ASCII characters. Non-ASCII bytes (>= 0x80) fall through to the existing
unicode-aware path unchanged.

## Throughput (Apple M4, 10 cores, in-memory benchmark)

| | MB/s | vs CPython serial |
|-|------|------------------|
| v0.6.1 serial | 43.9 | 4.3x |
| v0.6.2 serial | 53.6 | 5.3x |
| v0.6.1 parallel | 220.8 | 21.6x |
| v0.6.2 parallel | 240.3 | 23.5x |
| CPython serial baseline | 10.2 | 1x |

+22% on serial (the isolated optimization gain). The parallel wall-clock
number from disk is 262 MB/s (25.7x CPython).

## Implementation

Two tables replace the utf8 decode call for the common ASCII case:

- `identStartTable [128]bool`: true for `a-z`, `A-Z`, `_`
- `identContTable [128]bool`: true for `a-z`, `A-Z`, `0-9`, `_`

The hot loop in `scanNameOrPrefixedString` now reads one byte and checks
`identContTable[c]` before falling through to `utf8.DecodeRuneInString`.
The initial dispatch in `nextInternal` uses `identStartTable[c]` for the
identifier-start check.

No behavior change for any input.
