# v0.6.3: no-newline fast advance, comment IndexByte, sawNonASCII

Three redundant per-byte operations eliminated from the lexer hot paths.

## Changes

**Comment skip via `strings.IndexByte`** — `skipSpace()` and the
`nextInternal` indent-block loop now jump to the end-of-comment newline with
a single SIMD call instead of advancing byte by byte through comment text.

**No-newline fast advance** — ASCII identifier bytes and digit bytes can never
be `'\n'`, so the inner loops in `scanNameOrPrefixedString` and `scanNumber`
now do `s.off++; s.col++` directly, skipping the `'\n'` branch inside
`scanner.advance`.

**sawNonASCII flag** — instead of scanning every identifier a second time with
`hasNonASCII` to decide whether to call `norm.NFKC`, a boolean flag is set
during the scan whenever the non-ASCII code path is taken. `hasNonASCII` is
removed.

## Throughput (Apple M4, 10 cores, in-memory benchmark)

5×20 s runs, back-to-back comparison within the same session:

| | MB/s | vs CPython serial |
|-|------|------------------|
| v0.6.2 (same session) | 82.4 | 8.1x |
| v0.6.3 serial | 92.4 | 9.1x |
| v0.6.2 parallel | 256.6 | 25.2x |
| v0.6.3 parallel | 265.9 | 26.1x |
| CPython serial baseline | 10.2 | 1x |

+12% serial. The parallel gain is within measurement noise; the lexer
optimizations are most visible on the single-threaded path.
