# v0.6.4: per-parse arena allocator (sync.Pool + slice arena)

Every AST node was previously a separate heap allocation (`&Name{}`,
`&BinOp{}`, etc.), creating ~1175 GC-tracked objects per file. This release
replaces individual allocations with append-to-slice operations on a per-parser
arena, and reuses the pre-grown slices across ParseFile calls via sync.Pool.

## Implementation

**`parser/arena.go`** — 77 slice fields (one per AST node type) + generic helper:

```go
func arenaAlloc[T any](s *[]T, v T) *T {
    *s = append(*s, v)
    return &(*s)[len(*s)-1]
}
```

All 169 `&TypeName{...}` call sites in `parser.go` and `stmt.go` now use
`arenaAlloc(&p.ar.field, TypeName{...})`.

**`arena.reset()`** — truncates all 77 slices to length 0, keeping backing
arrays for the next parse.

**`stmtParserPool sync.Pool`** — ParseFile gets a parser from the pool, resets
its arena, parses, and puts it back. After the first file, each goroutine's
P-local pool slot holds a parser with pre-grown arenas. Subsequent parses
append to existing capacity with near-zero allocation.

**Safety**: the returned AST is valid until the next ParseFile call on the same
goroutine. Sequential per-file processing (parse → use → discard) satisfies
this automatically.

## Throughput (Apple M4, 10 cores, in-memory benchmark)

5×20 s serial + 3×20 s parallel, back-to-back against v0.6.3 baseline:

| | MB/s | vs CPython serial |
|-|------|------------------|
| v0.6.3 serial | 92.3 | 9.1x |
| v0.6.4 serial | 96.1 | 9.4x |
| v0.6.3 parallel | 276.9 | 27.1x |
| v0.6.4 parallel | 371.8 | 36.4x |
| CPython serial baseline | 10.2 | 1x |

+4% serial, +34% parallel. Memory per iteration: 186 MB → 70 MB (−62%).
Allocation count: 4.71M → 2.35M per iteration (−50%).

The parallel gain is large: reducing per-goroutine GC pressure lowers the
cross-goroutine GC coordination overhead that was limiting parallel throughput.
