# v0.4.2 — corpus parse fixes

Ten real parse failures found in the 18-package corpus are fixed. All 9011
corpus files now parse cleanly (7 remaining failures are intentionally invalid
test fixtures that CPython itself rejects).

## Fixes

- **String concatenation false-positive**: a plain string whose content starts
  with `b:` (e.g. `"b:<RPrimitive ...>"`) was incorrectly flagged as a bytes
  literal when used in implicit concatenation. The token now carries an explicit
  `strKind` field instead of relying on a `b:` prefix in the value string.

- **Unicode combining marks in identifiers**: `isIdentPart` now includes Mn
  (non-spacing mark) and Mc (spacing combining mark) Unicode categories, matching
  Python's UAX #31 `ID_Continue` property. Identifiers like `Q̇_per_meter`
  and `មុ` now parse correctly.

- **Inline `type` statement in class body**: `class X: type InClass = int` was
  rejected because the inline-body path (`parseSimpleStmtList`) did not check
  for type alias syntax. The check is now present.

- **Type param starred defaults**: `type D[*Ts=*int]` (PEP 696) was rejected
  because `parseTypeParamDefault` did not handle a leading `*`. The default is
  now parsed as `Starred(value=...)`, matching CPython.

- **`with` statement edge cases**: three forms were rejected:
  - `with (a), (b):` — parenthesized single items in multi-item with
  - `with (a, *b):` — starred element makes `(...)` a tuple context manager
  - `with (a for b in c):` — generator expression context manager

- **Backslash before indented block**: a lone `\` on a blank line inside a class
  or function body was not treated as a line continuation in the indent scanner.

- **For-loop target with general expression**: `for ((expr) or {})['a'] in x:`
  was rejected because `parseTargetAtom` used `parseTargetList` (targets only)
  for `(...)`. It now uses `parseParenAtom` to handle arbitrary expressions
  followed by postfix operators.

- **Pattern matching with soft-keyword args**: `case Cls(match=x, case=y):` and
  `case [a as match]:` were rejected because `match` and `case` were excluded
  from keyword-argument and `as`-capture positions in pattern matching.

- **CRLF files spurious DEDENT**: files with Windows `\r\n` line endings
  triggered false DEDENT tokens on blank lines because `\r` was not skipped in
  the indent computation loop.

- **Empty `Dict()` in py3.13+ ast dump**: `{}` was dumped as
  `Dict(keys=[], values=[])` instead of `Dict()` under the `show_empty=False`
  mode used by Python 3.13 and later.

## Test fixtures

Fixtures 107–114 added to `tests/grammar/` covering all fixed cases.
