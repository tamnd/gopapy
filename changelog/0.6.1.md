# v0.6.1: parallel file processing

`benchCmd`, `checkDir`, and `unparseCmd --check` now process files through a
worker pool sized to `runtime.NumCPU()`. Each file is parsed independently;
ParseFile has no shared mutable state, so no synchronization is needed inside
the hot path.

## Throughput (Apple M4, 10 cores)

| | MB/s | vs CPython serial |
|-|------|------------------|
| v0.6.0 serial | 69 | 6.8x |
| v0.6.1 parallel | 334 | 32.7x |
| CPython serial baseline | 10.2 | 1x |
| CPython parallel (10 processes) | 29.8 | 2.9x |

4.8x speedup from parallelism. Gap to 100x: 3x remaining.

## Implementation

Files are pre-read sequentially (walk + ReadFile), then distributed to
workers via a buffered index channel. Results are written into a pre-allocated
slice indexed by file position, so aggregation is lock-free. Error messages
for failed files are buffered per-goroutine and flushed after workers finish,
keeping output coherent without a mutex on stderr.

The race detector (`go test -race ./...`) shows no data races.

## Commands

- `gopapy bench DIR`: parse loop is parallel; wall time reflects all cores.
- `gopapy check DIR`: parse loop is parallel; FAIL lines may print out of walk
  order but the final count is correct.
- `gopapy unparse --check DIR`: round-trip check is parallel; DIFF/FAIL lines
  may print out of walk order but counts and exit code are correct.

## Other fixes

- `ParseFile` now strips the UTF-8 BOM and normalizes CRLF to LF before
  tokenizing, matching CPython's source preprocessing behavior.
- `gopapy check` skips `lib2to3/tests/data/` when checking a Python stdlib
  directory; those files are intentional Python 2 fixtures, not valid Python 3.
- `corpus/cpython_bench.py` now reports both serial and parallel
  (multiprocessing.Pool) CPython throughput for a fair side-by-side comparison.
