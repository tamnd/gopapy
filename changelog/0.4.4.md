# v0.4.4 — corpus-astdiff CI job (informational)

A new `corpus-astdiff` CI job compares gopapy's `dump` output against CPython's
`ast.dump()` on a random sample of corpus files. The job is informational
(continue-on-error: true). It becomes a hard gate in v0.4.9.

## How it works

`internal/oracle/corpus_diff.sh` takes two optional arguments: sample size
(default 100) and seed (default today's date as YYYYMMDD). It:

1. Walks corpus/src/, sorts the file list for reproducibility.
2. Uses Python's `random.shuffle` with the given seed to select the sample.
3. For each file, runs both `gopapy dump` and CPython `ast.dump()`.
4. Diffs the two outputs; prints MISMATCH lines and the diff to corpus_diff.log.
5. Exits non-zero if any file mismatches or errors.

The seed being date-based means each day's CI run covers a different slice of
the corpus. To reproduce a specific CI run, pass its date as the second
argument: `bash internal/oracle/corpus_diff.sh 100 20260428`.

## CI job

The `corpus-astdiff` job caches corpus/src under the same key as corpus-parse
and corpus-roundtrip (`corpus-src-${{ hashFiles('corpus/packages.txt') }}`).
The diff log is uploaded as a GitHub Actions artifact on every run so mismatch
diffs are visible without re-running locally.

## Known mismatches at tagging

AST dump parity is not yet complete. Known categories of mismatch seen in the
first local run:

- `Try` nodes: gopapy omits `handlers=[]` when the list is empty; CPython
  includes it under `show_empty=False` for Python 3.14.
- Other `show_empty` fields on various node types.

These will be fixed in a later release before corpus-astdiff becomes a hard gate.
