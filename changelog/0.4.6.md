# v0.4.6 — AST dump parity fixes

Four AST dump correctness bugs are fixed, bringing corpus-astdiff from 28 failures
down to 10. Fixture 119 covers the new grammar case.

## Fixes

- **Walrus operator in `with` statement**: `with (x := a, y := b):` is now
  correctly parsed as a single `With` item whose context expression is a `Tuple`,
  matching CPython. Previously each element was parsed as a separate `WithItem`,
  producing the wrong AST shape. Fixture 119 covers the three affected forms.

- **`\N{name}` Unicode name escapes**: String literals containing `\N{SNOWMAN}`
  (and all other named Unicode characters) are now resolved to the actual
  character at parse time, matching CPython. A 148,853-entry lookup table derived
  from Python's `unicodedata` module is embedded at build time via `//go:embed`.

- **Lone surrogate escapes**: `"\ud83d"` (a lone surrogate, U+D800–U+DFFF) is
  now encoded as a WTF-8 byte sequence in the parsed string value, so `pyRepr`
  can emit `\udXXX` escape notation that matches CPython's `ast.dump()` output.
  Previously `WriteRune` silently converted lone surrogates to U+FFFD.

- **`kind='u'` propagation in `JoinedStr`**: When an f-string group is prefixed
  by a `u`-string token (e.g. `u"" f"text {x}"`), `kind='u'` is now applied
  exactly to the first `Constant` in the resulting `JoinedStr`, only when the
  very first string token in the group carries the `u` prefix. Previously the
  flag could appear on the wrong segment or be omitted entirely.
